## What DOESN'T work:
- Abstract tables ("word=classification, number=regression" = "rata system")
- Colored box PPT slides (unreadable, "torture", "just rectangles")
- Repeating same explanation with different words
- Starting with model selection before teaching problem decomposition
- Assuming center=0.5 for eye gaze (every face is different)
- Manual calibration steps ("what if user doesn't do it?")
- Baby steps / over-explaining simple concepts

## What WORKS:
- "Watch the human" approach — decompose problems by observing what a human expert does
- The 3 Questions (Q1: Data? Q2: Thinking? Q3: Labels?)
- Building and running code immediately (not just theory)
- Rolling auto-calibration (no user action needed)
- Real exam hall scenarios and edge cases
- Student asks excellent engineering questions ("what if camera at different angle?", "what about multiple students?", "will it work deployed?")

## How to apply:
- Always start with practical implementation over theory
- Use "watch the human" for any new problem
- Don't create colored-box PPTs — use proper diagrams or skip slides
- When student says "not making sense" — change approach completely, don't rephrase
- Student thinks like an engineer — respect that by discussing real constraints
